# Monitoring & Observability Runbook

## Overview
This runbook covers monitoring stack incidents, alert fatigue, log pipeline failures, and observability gaps for InfraMind infrastructure. Stack: Prometheus, Grafana, CloudWatch, Fluent Bit, Jaeger.

---

## CRITICAL: Observability Error Quick Reference

| Symptom | Component | First suspect |
|---|---|---|
| Grafana dashboards show "No data" | Prometheus/datasource | Prometheus down, wrong datasource URL |
| Alerts not firing despite known issue | Alertmanager | Alertmanager down, alert silenced, wrong threshold |
| Too many false-positive alerts | Alert rules | Threshold too sensitive, missing `for` duration |
| Logs missing in Kibana/CloudWatch | Fluent Bit | Fluent Bit DaemonSet crashed, log path wrong |
| Traces not appearing in Jaeger | App / Jaeger | Tracing not instrumented, Jaeger collector down |
| Metrics lag by 5-15 minutes | Prometheus | Scrape interval too long, remote write backlog |

## Issue 1: Prometheus Not Scraping Targets

**Symptoms:**
- Grafana shows "No data" for metrics that were working before
- Prometheus targets page shows targets as DOWN
- Alerts not firing even though the issue exists

**Diagnosis:**
```bash
# Check Prometheus pod health
kubectl get pods -n monitoring -l app=prometheus
kubectl logs -n monitoring -l app=prometheus --tail=50

# Check Prometheus targets via port-forward
kubectl port-forward -n monitoring svc/prometheus 9090:9090
# Open: http://localhost:9090/targets
# Look for targets with "DOWN" state and read the error

# Common target errors:
# "connection refused" = service not running on the scrape port
# "context deadline exceeded" = scrape timeout (metrics endpoint too slow)
# "403 Forbidden" = metrics endpoint requires auth Prometheus doesn't have

# Check ServiceMonitor is configured correctly
kubectl get servicemonitor -n monitoring
kubectl describe servicemonitor <name> -n monitoring

# Verify the service has the right labels for Prometheus to discover it
kubectl get service <app-service> -o yaml | grep -A 5 labels
```

**Root Cause Analysis:**
- Application pod restarted and metrics port changed
- ServiceMonitor label selector doesn't match Service labels
- Metrics endpoint added auth requirement that Prometheus doesn't handle
- Prometheus RBAC — can't list pods/services in new namespace

**Immediate Fix:**
```bash
# Check Prometheus RBAC — needs ClusterRole with get/list/watch on pods, services, endpoints
kubectl get clusterrolebinding prometheus -o yaml

# Add namespace to Prometheus namespace selector if missing
kubectl edit prometheus -n monitoring
# Add to spec.namespaceSelector.matchNames: [<new-namespace>]

# Force Prometheus config reload
kubectl exec -n monitoring deployment/prometheus -- \
  curl -X POST http://localhost:9090/-/reload
```

## Issue 2: Alerts Not Firing (False Negatives)

**Symptoms:**
- Known issue exists (error rate high, pods down) but no alert received
- Alertmanager shows no active alerts
- On-call rotation not paged

**Diagnosis:**
```bash
# Step 1: Is Alertmanager running?
kubectl get pods -n monitoring -l app=alertmanager

# Step 2: Are alerts being generated by Prometheus?
kubectl port-forward -n monitoring svc/prometheus 9090:9090
# Open: http://localhost:9090/alerts
# Are the relevant alerts in FIRING or PENDING state?

# Step 3: Is Alertmanager receiving them?
kubectl port-forward -n monitoring svc/alertmanager 9093:9093
# Open: http://localhost:9093/#/alerts
# Check for active alerts and silences

# Step 4: Check if alert is silenced
kubectl exec -n monitoring deployment/alertmanager -- \
  amtool silence query --alertmanager.url=http://localhost:9093

# Step 5: Check Alertmanager routing config
kubectl get configmap alertmanager-config -n monitoring -o yaml | \
  grep -A 30 "route:"

# Step 6: Check notification receiver (Slack/PagerDuty) logs
kubectl logs -n monitoring deployment/alertmanager | grep -i "error\|fail\|notify"
```

**Root Cause Analysis:**
- Alert has `for: 10m` — issue resolved before alert promoted from PENDING to FIRING
- Alert silenced by someone during a previous incident who forgot to remove silence
- Alertmanager routing doesn't match alert labels — goes to wrong/null receiver
- Slack webhook URL rotated — Alertmanager still using old URL
- Alert threshold too high — error rate must be 50% before firing, but only got to 40%

**Immediate Fix:**
```bash
# Remove erroneous silence
kubectl exec -n monitoring deployment/alertmanager -- \
  amtool silence expire <silence-id> \
  --alertmanager.url=http://localhost:9093

# Test Alertmanager notification manually
kubectl exec -n monitoring deployment/alertmanager -- \
  amtool alert add \
  alertname=TestAlert severity=critical \
  --alertmanager.url=http://localhost:9093

# Update Slack webhook
kubectl create secret generic alertmanager-slack \
  --from-literal=webhook-url=<new-webhook-url> \
  -n monitoring --dry-run=client -o yaml | kubectl apply -f -
kubectl rollout restart deployment/alertmanager -n monitoring
```

## Issue 3: Alert Fatigue — Too Many False Positive Alerts

**Symptoms:**
- On-call receiving 50+ alerts per shift, most are noise
- Team starts ignoring alerts — dangerous
- Same alerts firing repeatedly for transient blips

**Diagnosis:**
```bash
# Find which alerts fire most often
kubectl exec -n monitoring deployment/alertmanager -- \
  amtool alert --alertmanager.url=http://localhost:9093 | \
  awk '{print $1}' | sort | uniq -c | sort -rn | head -20

# Check alert rules missing "for" duration (fires on first data point)
kubectl get prometheusrule -A -o yaml | \
  grep -B 5 "expr:" | grep -v "for:"
# Rules without "for:" fire immediately on any threshold breach
```

**Root Cause Analysis:**
- Alert rules missing `for` duration — single bad data point triggers alert
- Threshold set too low for normal variance
- Same condition generating multiple alerts (no grouping configured)
- Alerts not grouped by cluster/service — one cluster issue fires N alerts

**Fix — Improve alert rules:**
```yaml
# BAD — fires on any single breach
- alert: HighErrorRate
  expr: error_rate > 0.05

# GOOD — must be above threshold for 5 minutes consistently
- alert: HighErrorRate
  expr: error_rate > 0.05
  for: 5m          # Must be sustained
  labels:
    severity: warning
  annotations:
    summary: "Error rate above 5% for 5 minutes"
    runbook: "https://wiki/runbooks/application_errors"

# Add grouping in Alertmanager to reduce notification volume
route:
  group_by: ['alertname', 'cluster', 'namespace']
  group_wait: 30s       # Wait 30s for more alerts before notifying
  group_interval: 5m    # Send at most one notification per 5 min per group
  repeat_interval: 3h   # Re-notify if still firing after 3h
```

## Issue 4: Logs Missing from CloudWatch / Fluent Bit

**Symptoms:**
- Pod logs not appearing in CloudWatch Log Groups
- Fluent Bit DaemonSet pods crashing or restarting
- Log gap in CloudWatch — logs stop at a specific time

**Diagnosis:**
```bash
# Check Fluent Bit DaemonSet health
kubectl get daemonset -n kube-system fluent-bit
kubectl get pods -n kube-system -l app=fluent-bit

# Check Fluent Bit logs for errors
kubectl logs -n kube-system -l app=fluent-bit --tail=50 | \
  grep -i "error\|fail\|retry"

# Common errors:
# "Access denied" = Fluent Bit node role missing CloudWatch permissions
# "No such file or directory" = log path changed
# "Retrying" = CloudWatch endpoint unreachable

# Check if the CloudWatch Log Group exists
aws logs describe-log-groups \
  --log-group-name-prefix /aws/containerinsights/inframind

# Check for data in CloudWatch
aws logs filter-log-events \
  --log-group-name /aws/containerinsights/inframind/application \
  --start-time $(date -d '30 minutes ago' +%s000) \
  --limit 5
```

**Root Cause Analysis:**
- Fluent Bit node role missing `logs:PutLogEvents` or `logs:CreateLogStream`
- Log group doesn't exist — Fluent Bit can't create it without permission
- Fluent Bit DaemonSet on new nodes doesn't have updated config
- Container log path format changed (Docker vs containerd log format)

**Immediate Fix:**
```bash
# Add CloudWatch permissions to node role
aws iam attach-role-policy \
  --role-name <eks-node-role> \
  --policy-arn arn:aws:iam::aws:policy/CloudWatchAgentServerPolicy

# Restart Fluent Bit DaemonSet to pick up new permissions
kubectl rollout restart daemonset/fluent-bit -n kube-system

# Manually create log group if auto-creation is failing
aws logs create-log-group \
  --log-group-name /aws/containerinsights/inframind-cluster/application
aws logs put-retention-policy \
  --log-group-name /aws/containerinsights/inframind-cluster/application \
  --retention-in-days 30
```

## Issue 5: Grafana Dashboard Shows Stale / No Data

**Symptoms:**
- Grafana shows "No data" on panels
- Dashboard shows data up to a point then flatlines
- "datasource not found" error in Grafana

**Diagnosis:**
```bash
# Test Prometheus datasource in Grafana
# Grafana → Configuration → Data Sources → Prometheus → Test

# Check if Prometheus is reachable from Grafana
kubectl exec -n monitoring deployment/grafana -- \
  curl -s http://prometheus:9090/-/healthy
# Expected: Prometheus is Healthy

# Check Grafana logs
kubectl logs -n monitoring deployment/grafana | \
  grep -i "error\|datasource\|fail" | tail -30

# Check Prometheus has recent data
kubectl port-forward -n monitoring svc/prometheus 9090:9090
# Query: up{} — should show 1 for all targets
# Query: prometheus_tsdb_head_samples_appended_total — should be increasing
```

**Root Cause Analysis:**
- Prometheus URL in Grafana datasource is wrong after service rename
- Prometheus storage full — stopped ingesting new data
- Grafana session expired — refresh the page (common non-issue)
- Time range set to future in Grafana — no data exists yet

**Immediate Fix:**
```bash
# Check Prometheus storage
kubectl exec -n monitoring deployment/prometheus -- \
  df -h /prometheus

# If storage full — increase PVC
kubectl patch pvc prometheus-storage -n monitoring \
  -p '{"spec":{"resources":{"requests":{"storage":"100Gi"}}}}'

# Or reduce retention to free space
kubectl edit prometheus -n monitoring
# Set: retention: 15d (down from 30d)
```

## Observability Health Check
```bash
# Full monitoring stack status
kubectl get pods -n monitoring
kubectl get pods -n kube-system | grep -E "fluent|metrics-server"

# Prometheus health
kubectl port-forward -n monitoring svc/prometheus 9090 &
curl http://localhost:9090/-/healthy
curl http://localhost:9090/-/ready

# Check metrics ingestion rate (should be > 0 and stable)
curl -s "http://localhost:9090/api/v1/query?query=rate(prometheus_tsdb_head_samples_appended_total[5m])"

# Alert summary
kubectl exec -n monitoring deployment/alertmanager -- \
  amtool alert --alertmanager.url=http://localhost:9093

# Check CloudWatch log ingestion (recent logs present?)
aws logs describe-log-streams \
  --log-group-name /aws/containerinsights/inframind-cluster/application \
  --order-by LastEventTime --descending --limit 5
```
